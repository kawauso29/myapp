Ledger V2 詳細設計書
====================

作成目的
--------

本設計書は、既存のLedger系システムをv2として作り直すための詳細設計である。

v2の目的は、v1の機能をそのまま再実装することではない。
v1で得た知見、失敗、過剰な自動化、必要だった台帳構造を整理し、
最初から「制御・監査・停止・レビュー」を中心にしたLedger Kernelを作ることである。

v1は仕様発掘用プロトタイプとして扱う。
v2は本運用を前提にした最小・安全・拡張可能なLedger基盤として設計する。

基本方針
--------

1. v1の規模・思想は否定しない
2. ただしv2は最小カーネルから作る
3. v1のコードをそのまま移植しない
4. v1から移植するのは仕様・知見・ドメイン概念である
5. v2では最初に制御層を作る
6. すべての処理はRunとして記録する
7. すべての副作用はEventとして記録する
8. すべての自動起票はcanonical_keyで重複排除する
9. すべての成果物は人間レビュー前提にする
10. すべてのRunnerはdry_runできる
11. すべてのRunnerはStopCondition / CircuitBreakerで停止できる
12. 人間承認なしに戦略・設定・本番状態を変更しない
13. Trading / HR / OrgChange / Portfolioは初期接続しない
14. Monthly / Quarterly / Annualは初期実装しない
15. AutoMergeはv2機構安定まで一時停止する（安定後は自動マージ・自動デプロイへ移行する）

v1とv2の位置づけ
----------------

v1:
- 実験炉
- 仕様発掘用プロトタイプ
- 台帳・Runner・自動化の試行錯誤結果
- readonlyまたはfreeze対象
- v2のための参照元

v2:
- Ledger Core Kernel
- 本運用を前提とする最小基盤
- 監査可能
- 停止可能
- dry_run可能
- 人間レビュー前提
- 拡張は段階昇格制

基本判断:

v1 = freeze
v2 = run

v1は即削除しない。
ただしv2評価中にv1のRunnerや自動化が同時に動くと結果が濁るため、
v1の自動実行は停止する。

v2の設計思想
------------

v2のLedgerは「完全自律意思決定OS」ではない。

初期のLedger V2は以下の役割に限定する。

- 観測する
- 異常を検知する
- Ticket候補を作る
- 成果物draftを作る
- 人間レビューへ回す
- 実行結果を記録する
- 停止条件を評価する
- 改善効果を測る
- ノイズ率を測る
- 採用率を測る

v2初期では、Ledgerが直接戦略を変更したり、設定を変更したり、本番に影響する変更を確定してはいけない。

最終的な定義:

Ledger V2は、自律的に意思決定するOSではなく、
人間がより速く・一貫して意思決定するための
観測・起票・記録・停止・監査・レビュー基盤である。

- skipped

canonical_key設計:

canonical_keyは意味的重複を防ぐためのキーである。

例:

source_type + source_id + metric_name + anomaly_type + period_bucket

具体例:

ai_sns:global:post_count:low:2026w18
ci:main:success_rate:low:2026w18
ai_sns:dm:reply_rate:low:2026w18

制約:

open / in_progress / deferred 状態のcanonical_keyは一意にする。

実装例:

unique partial index:
canonical_key unique where status in ('open', 'in_progress', 'deferred')

重要ルール:

- idempotency_keyだけに頼らない
- 同じ意味のTicketはcanonical_keyで抑止する
- resolved / rejected後の再起票は許可してよい
- 再起票する場合はprevious_ticket_idを持つ
- duplicateになった場合はduplicate_of_idを持つ

重要指標:

Artifactが承認・採用された割合。

runner_failure_rate:
Runner実行失敗率。

unresolved_ticket_age_avg:
未解決Ticketの平均滞留時間。

human_intervention_rate:
人間が修正・却下・停止した割合。

kpi_improvement_after_ticket_rate:
Ticket起票後にKPIが改善した割合。

stop_trigger_count:
StopCondition / CircuitBreakerが発火した回数。

duplicate_prevented_count:
重複Ticketを防いだ回数。

重要ルール:

- Ledger V2自身を評価対象にする
- 価値が出ているかを数値で見る
- Artifact採用率とTicketノイズ率を最重要視する

Service設計
-----------

LedgerV2::RunExecutor
---------------------

目的:
すべてのRunnerの統一入口。

責務:

- FeatureFlag確認
- CircuitBreaker確認
- Run作成
- idempotency制御
- dry_run制御
- 実行時間計測
- 成功・失敗・skip・blocked記録
- Event作成
- 例外処理
- 結果集計

呼び出し例:

LedgerV2::RunExecutor.call(:daily_runner, dry_run: true)
LedgerV2::RunExecutor.call(:weekly_runner, triggered_by: current_user.id)

禁止:

LedgerV2::DailyRunner.call を直接呼ぶこと。

疑似コード:

class LedgerV2::RunExecutor
  def self.call(runner_name, dry_run: false, triggered_by: nil, **args)
    new(runner_name, dry_run:, triggered_by:, args:).call
  end

  def call
    return create_skipped_run(:feature_disabled) unless Flags.enabled?(runner_name)
    return create_blocked_run(CircuitBreaker.reason_for(runner_name)) if CircuitBreaker.blocked?(runner_name)

    run = Run.create!(runner_name:, status: :running, dry_run:, triggered_by:, started_at: Time.current)

    result = runner_class.call(run:, dry_run:, **args)

    run.update!(
      status: :success,
      finished_at: Time.current,
      duration_ms: calculate_duration,
      created_ticket_count: result.created_ticket_count,
      created_artifact_count: result.created_artifact_count,
      duplicate_prevented_count: result.duplicate_prevented_count
    )

    result
  rescue => e
    run&.update!(status: :failed, error_class: e.class.name, error_message: e.message, finished_at: Time.current)
    raise
  end
end

LedgerV2::Flags
---------------

目的:
Ledger V2機能の有効・無効を管理する。

初期実装:

Rails configでよい。

例:

config.x.ledger_v2_flags = {
  daily_runner: true,
  weekly_runner: true,
  health_snapshot: true,
  ticket_creation: true,
  artifact_generation: true,
  monthly_runner: false,
  quarterly_runner: false,
  annual_runner: false,
  auto_pr: false,
  auto_merge: false
}

将来:

DB管理に移行してもよい。
ただし変更履歴をEventとして残す。

重要ルール:

- 初期値は保守的にする
- 新機能はデフォルトfalse
- 本番影響があるものは人間承認必須

LedgerV2::CircuitBreaker
------------------------

目的:
実行前に「今そのRunnerを動かしてよいか」を判定する。

判定条件:

- active StopConditionがある
- 前回同Runnerが失敗している
- 未レビューArtifactが多すぎる
- open Ticketが多すぎる
- duplicate候補が多すぎる
- CIが失敗中
- dry_runでないのに本実行許可がない
- 手動承認待ちが残っている

返す情報:

- blocked? true / false
- reason
- severity
- blocking_condition_id

重要ルール:

- CircuitBreakerはRunnerの前に必ず通す
- blockedになったこともRunとEventに記録する
- StopConditionがactiveなら必ず止める

LedgerV2::DailyRunner
---------------------

目的:
日次観測と最小限の異常検知を行う。

責務:

- MetricSnapshotを作る
- DetectMetricAnomaliesを呼ぶ
- Ticket候補を作る
- TicketDeduplicatorで重複確認する
- 必要ならTicketを作る
- Eventを残す

やらないこと:

- Artifactを大量生成しない
- 戦略変更しない
- 設定変更しない
- PRを作らない
- AI人格や記憶を変えない

初期対象KPI:

- AI-SNS投稿数
- AI-SNS DM数
- AI-SNS反応数
- エラー件数
- CI成功率
- open Ticket数
- pending Artifact数

出力:

- MetricSnapshot
- anomaly_detected Event
- Ticketまたはticket_deduplicated Event
- Run結果

LedgerV2::WeeklyRunner
----------------------

目的:
open Ticketを週次レビューし、Artifact draftを作る。

責務:

- open / deferred Ticketを集める
- 優先度を整理する
- 週次レビューArtifactを作る
- 改善候補をまとめる
- carry_over候補を記録する
- 人間レビュー待ちにする

やらないこと:

- 戦略を確定しない
- Ticketを大量に新規作成しない
- 自動PRを作らない
- 自動マージしない
- Monthly / Quarterlyへ勝手に昇格しない

出力:

- weekly_review Artifact
- ticket_updated Event
- artifact_created Event
- pending Review

LedgerV2::DetectMetricAnomalies
-------------------------------

目的:
MetricSnapshotから異常を検出する。

初期は単純な閾値でよい。

例:

- 投稿数が最低値未満
- DM数が急減
- エラー件数が閾値超過
- CI成功率が閾値未満
- open Ticket数が増えすぎ
- pending Artifact数が増えすぎ

重要ルール:

- 初期から高度なAI判定をしない
- 閾値・ルールベースから始める
- 判定理由をpayload_jsonに残す
- 異常検知だけで副作用を起こさない
- Ticket作成はOpenTicketに任せる

LedgerV2::OpenTicket
--------------------

目的:
異常や改善候補からTicketを作る。

責務:

- canonical_key生成
- TicketDeduplicator呼び出し
- Ticket作成
- duplicate時のEvent作成
- dry_run時の仮結果作成

重要ルール:

- canonical_keyなしの自動Ticket作成は禁止
- duplicate時はTicketを作らずEventに残す
- dry_run時はDB書き込みを避けるか、dry_run Run配下にのみ記録する

LedgerV2::TicketDeduplicator
----------------------------

目的:
意味的に同じTicketの重複を防ぐ。

判定順:

1. canonical_key完全一致
2. source_type + source_id + metric_name + anomaly_type一致
3. 近いperiod_bucketの類似Ticket
4. statusがopen / in_progress / deferredか確認
5. resolved / rejectedなら再起票可能

出力:

- duplicate? true / false
- existing_ticket
- reason
- duplicate_level

重要ルール:

- 完全一致は必ず止める
- 類似一致は初期ではwarning Eventに留めてもよい
- 将来的にAI類似判定を入れてもよいが、初期は入れない

-----------------------------

目的:
WeeklyRunner用のArtifact本文を作る。

入力:

- run

- metric snapshots

- health snapshot

出力:

- Artifact draft body

内容:

- 今週の異常
- open Ticket一覧
- 改善候補
- carry_over

- StopCondition候補
- ノイズ候補

重要ルール:

- 確定調で書きすぎない
- 人間判断が必要な箇所を明示する
- 自動で施策確定しない

LedgerV2::PublishArtifact
-------------------------

目的:
Artifactを保存し、Review待ちにする。

重要ルール:

- 初期statusはdraftまたはpending
- 人間承認なしにpublishedにしない
- Reviewレコードを作成してもよい
- Artifact作成Eventを残す

LedgerV2::EvaluateImprovement
-----------------------------

目的:
Ticket作成後にKPIが改善したかを見る。

判定例:

- Ticket作成前後のMetricSnapshot比較
- エラー件数が減ったか
- 投稿数が回復したか
- CI成功率が改善したか
- open Ticket数が減ったか

出力:

- improvement_detected Event
- improvement_not_detected Event
- close candidate
- carry_over candidate

重要ルール:

- 自動closeは初期ではしない
- close候補を出して人間レビューに回す
- 改善が見えない場合はTicketをdeferred候補にする

LedgerV2::CalculateHealthSnapshot
---------------------------------

目的:

Runner実行フロー
----------------

DailyRunnerのフロー:

1. RunExecutor.call(:daily_runner)
2. Flags.enabled?(:daily_runner)
3. CircuitBreaker.blocked?(:daily_runner)

5. MetricSnapshot作成
6. anomaly検出

8. duplicate判定

1. RunExecutor.call(:weekly_runner)

3. CircuitBreaker.blocked?(:weekly_runner)
4. Run作成
5. open Ticket収集
6. MetricSnapshot収集
7. HealthSnapshot参照
8. Weekly Artifact draft作成
9. Review pendingにする
10. Run成功記録
11. Event作成

dry_run設計
-----------

v2ではdry_runを必須機能とする。

dry_runの目的:

- 本番DBへの副作用を避ける
- 何が起きる予定か確認する
- 管理画面で差分を見る
- 新Runnerを安全に試す

dry_runで許可するもの:

- Runをdry_runとして記録
- Eventをdry_runとして記録
- 仮Ticket結果の表示
- 仮Artifact bodyの表示
- duplicate判定の表示

dry_runで禁止するもの:

- 通常Ticketの作成
- 通常Artifactの作成
- 外部APIへの書き込み
- PR作成
- 設定変更
- StopCondition解除
- 本番状態変更

実装方針:

- dry_runでもRunは残す
- dry_run結果はmetadata_jsonに保存してよい
- dry_runによる仮Artifactは保存するならreview対象外にする

権限設計
--------

初期はadmin限定でよい。

権限レベル:

viewer:
- Run / Event / Ticket / Artifact / Healthを閲覧できる

operator:
- dry_run実行できる
- Ticketをdeferできる
- Artifactをreviewできる

admin:
- Runnerを本実行できる
- StopConditionを作成・解除できる
- FeatureFlagを変更できる

owner:
- critical StopConditionを解除できる
- v2機能をCoreへ昇格できる
- 本番影響のある設定を変更できる

重要ルール:

- AIによるFeatureFlag変更は禁止
- AIによるStopCondition緩和は禁止
- 本番影響のある操作はadmin以上
- critical解除はowner限定

AI-SNS連携設計
---------------

初期連携はreadonlyから始める。

Phase A: Readonly

Ledger V2がAI-SNSから読むもの:

- 投稿数
- DM数
- 反応数
- エラー件数
- active AI数
- active user数
- moderation候補数

Ledger V2がAI-SNSへ書くもの:

なし

Phase B: Proposal only

Ledger V2がAI-SNSへ出すもの:

- 改善提案Artifact
- モデレーション候補
- 投稿頻度調整案
- DM応答改善案

ただし、実際の変更は人間承認後。

Phase C: Approved write

人間承認後のみ限定的な書き込みを許可。

許可候補:

- 投稿頻度設定
- 軽微な表示設定
- moderation queueへの追加

禁止:

- AI人格の自動変更
- AI記憶の自動変更
- AI関係性の自動変更
- ユーザー影響の大きい設定変更
- 課金・権限・削除系操作

v1からの移行方針
----------------

移行するもの:

- 必要だったLedger概念
- KPI候補
- Ticket分類
- Artifact種類
- Stop条件の知見
- Runner失敗パターン
- 重複起票パターン
- 管理画面に必要な表示項目

移行しないもの:

- v1 Runnerの実装
- v1の巨大サービスオブジェクト
- v1の自動PR/自動マージロジック
- v1のAnnual / Quarterly自動実行
- v1のHR / OrgChange / Portfolio自動判断

移行手順:

1. v1をfreezeする
2. v1 inventoryを作る
3. v1の実行経路を止める
4. v1の有用概念をdocs/ledger_v2_requirements.mdに抽出する
5. v2 Kernelを作る
6. v2をdry_runで動かす
7. v2のDaily / Weeklyだけ本実行する
8. v1はreadonly参照元にする
9. v2が安定したらv1をarchive扱いにする

v1 freeze条件:

- v1 Runnerの自動実行停止
- v1 auto merge停止
- v1 auto PR停止またはdraft only
- v1 scheduler停止
- v1から本番状態を変える経路を停止

--------

- 7日以上安定稼働

- 重複Ticketが抑止されている
- Artifact acceptance rateが一定以上
- Ticket noise rateが高すぎない
- StopConditionが正常に効く

最終結論
--------

Ledger V2は、v1の単純な作り直しではない。

v1で膨らんだ構想を踏まえつつ、
v2では以下を最初から中心に置く。

- Run
- Event
- Ticket
- Artifact
- Review
- StopCondition
- MetricSnapshot
- HealthSnapshot

この8つがLedger V2 Kernelである。

v2初期に必要なのは、賢いAI運営ではない。
必要なのは、確実に記録され、止められ、レビューでき、重複せず、価値を測れる基盤である。

Daily / Weeklyの最小改善ループが安定するまで、
Monthly / Quarterly / Annual / HR / OrgChange / Portfolio / AutoMergeは持ち込まない。

v2の成功は、機能数ではなく以下で判断する。

- 重複を防げたか
- ノイズを減らせたか
- Artifactが使われたか
- Runnerが止められたか
- 人間が判断できたか
- KPI改善につながったか
- 何が起きたか説明できるか

これを満たしたLedger V2だけが、最終的に大きな運営OSへ拡張する価値を持つ。
