# 10. 過去事故と対策(ねむ犬プロジェクトから継承)

このドキュメントは Copilot Coding Agent が企画時に必ず参照する。
LINEスタンプ生成パイプラインで実際に発生した事故と、その構造的対策。

---

## 事故1: 背景透過で白い体まで抜ける

### 何が起きた
Canva 等の汎用背景削除アルゴリズムを使ったら、白背景前提のために **白い犬の体まで透過** されてしまった。

### 構造的原因
背景削除AIは「最も外側の領域」を背景とみなす。白い体と白い背景の境界が曖昧で、ピクセル単位で「これは背景」「これは体」を分離できない。

### 対策
- 生成時は **単色グリーン背景(グリーンバック)** に統一(キャラの白と色域が遠い)
- 透過処理は HSV 色域マスク(緑色相のみを抜く)
- mini_magick 採用後: `-fuzz 25% -transparent green` で実装(白は彩度0なので絶対残る)

### 適用範囲
- 生成プロンプトに「背景: 単色グリーン」を必ず明記
- スピル抑制も併用(緑チャンネル抑制)

## 事故2: ベース段階でパック8枚化される

### 何が起きた
ベース画像生成時に、AI が先走って「8枚のスタンプ一覧」を作ってしまった。

### 構造的原因
プロンプト一式(00 + 01 + 02 + 03)を全部渡してたため、AI が「最終形」を勘違いして作る。

### 対策
**Phase 分割**(段階渡し):
- Phase 1 (Brand base): brand_theme + base のみ → ベース画像
- Phase 2 (Pack sheet): + pack 情報 → 8枚一覧
- Phase 3 (個別 stamp): pack 情報のみ + 参照画像 → 1枚ずつ

Rails 実装では **PromptComposer** が Phase 別に異なるプロンプトを合成する。
管理画面でそれぞれを別フィールドに表示し、原田さんが Designer に Phase ごとに貼る。

### 適用範囲
- `02_base.md` の「強制プロンプト」には**パック情報を入れない**
- `03_stamp_pack.md` の「個別書き出しルール」には**ベース仕様を入れない**(参照画像で渡す)
- `PromptComposer#compose_for_brand_base` は `01_brand_theme.md` + `02_base.md` のみ参照

## 事故3: 実行書(共通ガイド)が画像化される

### 何が起きた
全ファイル共通の「実行ガイド」文面が AI の入力に混じっていて、AI がそれを画像テーマと勘違いして「開発ガイド風の図解」を生成した。

### 構造的原因
プロンプトの中に「あなたへの指示文」と「画像にする内容」が区別なく混在していた。

### 対策
- システム指示(画像化禁止)は別ファイル / 別レイヤーに切り出す
- 画像化対象のテーマ部分だけを `PromptComposer` が抽出して渡す
- 必要なら negative_prompt に「text in image about workflows」を含める

### 適用範囲
- `PromptComposer` が md を丸投げしない、必要セクションだけ抽出
- `01_brand_theme.md` の `## OKな方向` セクションだけを使う等の絞り込み

---

## 事故4: 「禁止」設計で派生パックを潰す

### 何が起きた
最初に「雲・星は禁止」と設定したら、後から Dreamy Comfort 系の派生パックを作ろうとした際にプロンプトの整合性が取れなくなった。

### 構造的原因
「禁止リスト」は **将来の派生パック余地を潰す**。

### 対策
「禁止」ではなく **「今回採用しない」** 表現を使う。表現レイヤー(Core / Work / Dream)を定義し、レイヤーごとに採用要素を切り替える。

### 適用範囲
- `01_brand_theme.md` の `## 表現レイヤー` で Core/Work/Dream を必ず分離
- `03_stamp_pack.md` の `## 採用しない要素(派生パックへの含み)` に書く(完全否定ではない)

---

## 事故5: 個別書き出しで顔・線・文字が揺れる

### 何が起きた
8枚一覧シートを作ってから個別書き出しに移ると、目が丸くなったり線が変わったり、別キャラに見えた。

### 構造的原因
個別生成プロンプトに「このキャラの全仕様」を埋め込んでも、AI の解釈で微妙にブレる。

### 対策
- 個別生成は **必ず参照画像(パックシート)を Designer に添付** する
- プロンプトに「参照画像と完全に同じ絵柄で再現」と明記
- パックシートが無い段階では brand.base_image を代替参照に

### 適用範囲
- `PromptComposer#compose_for_stamp` で参照画像のパスを言及する
- 03_stamp_pack.md の個別書き出しルールに「参照固定」を必ず書く
- 管理画面の stamp 詳細でも「Designer に貼る時は pack.sheet_image を必ず添付」と注記

---

## 事故6: 漢字崩れ

### 何が起きた
個別書き出し時に漢字が崩れた(誤字・似た文字に置換・架空漢字)。

### 構造的原因
画像生成 AI は日本語フォント描画が苦手。特に漢字。

### 対策
- プロンプトに「漢字は正しく丁寧に描く」を明記
- 「崩れたら再生成、ひらがなに逃げない」をルール化
- negative_prompt に「wrong kanji, text errors, garbled text」を含める
- 必要なら label は**漢字が少ない表記**を選ぶ(brand_theme で「漢字は読みやすいものを優先」)

### 適用範囲
- `02_base.md` の「最重要ルール(文字)」セクションに必ず明記
- `manifest.yml` の label は 5〜10文字推奨(短いほど崩れにくい)

---

## 事故7: パック企画が前パックと被る

### 何が起きた
同じブランドで複数 Pack を作るうちに、シチュエーションが既存と重複してテーマが薄まった。

### 構造的原因
Copilot が既存パックを参照せず、ゼロから企画した。

### 対策
- Pack 企画 Issue で「既存 packs を必ず確認、重複しない」を明示
- 企画完了条件チェックリストに「既存 pack と series_theme が被らない」「label が既存と重複しない」を含める

### 適用範囲
- `08_PLANNING_GUIDE.md` の Pack 企画手順に明記済
- 必要なら Rails 側で `manifest.yml` の重複検出を `BrandSourcesSyncer` で実装

---

## 適用方針まとめ

| 過去事故 | 対策の所在 |
|---|---|
| 1. 白い体が抜ける | `ChromaKeyProcessor` (HSV+fuzz) |
| 2. ベース段階でパック化 | Phase 分割(別Job) |
| 3. ガイド画像化 | `PromptComposer` のセクション抽出 |
| 4. 禁止で派生潰し | brand_theme.md の表現レイヤー設計 |
| 5. 個別書き出しで揺れ | img2img で sheet_image 参照 |
| 6. 漢字崩れ | 02_base.md の文字ルール + negative_prompt |
| 7. パック重複 | PLANNING_GUIDE のチェックリスト |

これらは **Copilot が企画時に意識すべきこと** と **Rails が画像生成時に守ること** の両方を含む。
